# ADR 0013: OSIF — Open Supply-chain Incident Format

**Status:** Accepted (spec proposal), 2026-07-14 · **Author:** vetlock, but the spec is CC0 and community-governed.

## Context

CVE, GHSA, and OSV tell you that *a package version is bad*. They do not
capture:

- The **entry point** the attack used (install-script? bundled dep? `.pth`
  file? typosquat?).
- The **capabilities** the malicious code gained (net-egress? secret-read?
  fs-write to `.ssh`? crypto-mine?).
- The **evasion techniques** the attacker used (obfuscation, dataflow
  laundering, `Reflect.get` alias, base64-decoded URLs).
- How the payload was **delivered through the dependency graph** (direct,
  transitive at depth-N, bundled inside another package, workspace-linked).
- Which signals a scanner would use to **detect** it.

The industry has no shared, machine-readable vocabulary for those mechanics.
Every vendor re-describes each incident ad hoc, and defenses cannot be
compared apples-to-apples. **OSIF is that vocabulary.**

## Decision

Propose OSIF as a **neutral, open, community-governed** JSON-schema for
describing the mechanics of a supply-chain incident. Specifically:

- **License:** CC0 (public domain equivalent). Not Apache, not MIT — the
  vocabulary must be usable by anyone including competitors. This is the
  point.
- **Governance:** proposed by vetlock, but immediately offered to OpenSSF /
  peer maintainers for adoption. The spec lives in its own repo
  (`vetlock-dev/osif-spec` at first, transferred to a neutral home the moment
  a neutral home exists).
- **Branding:** the spec is NOT called "vetlock incident format" and is NOT
  vetlock-licensed. vetlock is the *first mover*, not the *owner*.
- **Interop-not-replace:** OSIF interoperates with OSV (which it can be
  embedded in as an extension) and cites GHSA/CVE/OpenSSF taxonomies where
  they exist. It does not attempt to replace them.

### Why this is the keystone artifact (packet §3.8)

OSIF's `entry_point[]` and `capabilities_gained[]` fields ARE the same
enumeration as the CAPABILITY-MAP taxonomy from ADR 0011 (completeness
doctrine). Concretely:

- The `entry_point` values are exactly the `class: entry-point` entries in
  `packages/detectors/src/capability-map.json`.
- The `capabilities_gained` values are exactly the `class: sink` entries.

This is enforced by a test — `osif-taxonomy-matches-capability-map.test.ts`
in the spec repo (see enforcement below). One taxonomy, three consumers:
- The doctrine (internal law that says every enumerated class must have a
  detector).
- The benchmark (public scoreboard scored against OSIF-described attacks).
- OSIF (the shared industry vocabulary).

Owning the vocabulary is the same category-defining move as owning the
benchmark. It's a public good that also compounds our credibility.

## Schema (v0.1 — draft, iterate before public proposal)

```jsonc
{
  "$schema": "https://osif.dev/schema/v0/incident.json",
  "id": "OSIF-2018-EVENT-STREAM",             // Stable id, human-readable
  "ecosystem": "npm",                          // npm | pypi | golang | crates | gems | maven
  "package": "flatmap-stream",                 // Canonical package name in that ecosystem
  "affected_versions": ["0.1.1", "0.1.2"],    // Explicit list OR a semver range string

  "entry_point": "transitive-injection",       // Enum from CAPABILITY-MAP
  "entry_point_details": {
    "parent_package": "event-stream",
    "parent_version": "3.3.6",
    "depth_from_root": 2
  },

  "capabilities_gained": [                     // Enum-list from CAPABILITY-MAP
    "code-execution",
    "secret-read",
    "net-egress"
  ],

  "evasion_techniques": [                      // Enum-list, extensible
    "obfuscation-aes",
    "environment-scoped-only-in-copay-wallet"
  ],

  "delivery": {
    "graph_paths": [
      ["your-app", "some-dep", "event-stream", "flatmap-stream"]
    ],
    "publish_channel": "npm-registry",
    "publish_actor": "right9ctrl (new maintainer, backdoored release)"
  },

  "indicators": [                              // DEFANGED — enforced by the validator
    {
      "kind": "url",
      "value": "https://copayapi.example.invalid/api/…"   // .invalid TLD required for URL indicators
    },
    {
      "kind": "file",
      "value": "flatmap-stream/index.min.js.map"          // Path relative to package root
    }
  ],

  "timeline": {                                // ISO-8601 dates only
    "published": "2018-09-09",
    "discovered": "2018-11-20",
    "pulled_from_registry": "2018-11-27"
  },

  "detectability": {
    "signals": [                               // Detector IDs from CAPABILITY-MAP
      "install.script-added",
      "net.encoded-endpoint",
      "code.dynamic-loading-added"
    ],
    "notes": "Original payload was AES-encrypted; base64-decoded URL is caught by net.encoded-endpoint."
  },

  "references": [                              // CVE/GHSA/OSV + human writeups
    "CVE-2018-1000536",
    "GHSA-mh63-6h87-95cp",
    "https://blog.npmjs.org/post/180565383195/details-about-the-event-stream-incident"
  ],

  "osif_version": "0.1"                        // Schema version this document validates against
}
```

**Defang enforcement:** the JSON-schema validator has a `format: "defanged-url"`
check that fails validation when any URL indicator lacks an RFC-2606-reserved
TLD (`.invalid`, `.example`, `.test`, `.localhost`) OR looks like a real host.
This is *the* differentiator from a real-CVE report — OSIF documents describe
mechanics for defenders, not addresses for attackers.

### Enumerations (draft — the source of truth is CAPABILITY-MAP)

**`entry_point`** (initial list; extensible by PR):
- `install-script` — `postinstall`, `preinstall`, `prepare`, `prepublish`, `prepack`
- `main-import-side-effect` — top-level code in `main`/`module`/`exports` that runs on import
- `bin-executable` — code in a `bin`-declared script
- `bundled-payload` — malicious `node_modules/*` inside the package tarball
- `native-gyp-build` — `node-gyp` compilation step
- `transitive-injection` — a dep-of-dep introduced the payload (event-stream case)
- `maintainer-takeover` — same package name, new attacker-controlled release
- `typosquat` — near-name imitation of a legitimate package
- `git-source` — package resolved to a `git+` URL rather than the registry
- `file-source` — package resolved to a `file:` URL
- `alias-shadow` — yarn/pnpm `npm:` alias resolving to a different real package
- `integrity-tamper` — same version, different tarball bytes (registry compromise)
- `dot-pth` (PyPI) — `.pth` files executing at interpreter startup
- `build-hook` (PyPI) — `setup.py` / `pyproject.toml` build-time code

**`capabilities_gained`** (initial list; same as CAPABILITY-MAP sink classes):
- `code-execution`, `net-egress`, `fs-write`, `fs-read`, `secret-read`,
  `persistence`, `obfuscation-decode`, `crypto-mine`, `clipboard`,
  `process-enumeration`

**`evasion_techniques`** (initial list; extensible):
- `obfuscation` (minification, base64/hex encoding, rot-13 char arithmetic)
- `dynamic-dispatch` (`Reflect.get`, computed property access, `require.call`)
- `dataflow-laundering` (`x = 'ex'+'ec'; child_process[x]()`)
- `environment-scoped-only-in-<context>` (payload only fires in a specific
  wallet / hostname / user)
- `payload-encrypted-with-target-key`
- `bundled-inside-untrusted-transitive`
- `native-binary-payload`
- `wasm-payload`

## Enforcement

- **`osif-spec` repo** (to be created; see LAUNCH-CHECKLIST.md):
  - `schema/v0/incident.json` — the JSON-schema.
  - `validator/index.js` — a small JS lib that validates + reports.
  - `docs/SPEC.md` — human-readable spec.
  - `examples/` — one OSIF doc per historical attack, defanged.
  - `LICENSE` — CC0.
- **`osif-schema-valid.test`** (in `osif-spec`) — every example under
  `examples/` validates against `schema/v0/incident.json`; validator refuses
  documents with non-defanged URL indicators.
- **`osif-taxonomy-matches-capability-map.test`** (in `OJ-Uday/vetlock`) —
  reads OSIF's `entry_point` and `capabilities_gained` enumerations and asserts
  every value appears in `packages/detectors/src/capability-map.json` (and
  vice versa for the CAPABILITY-MAP classes marked "exposed to OSIF").
- **`osif-export-roundtrip.test`** — the `vetlock diff --format osif`
  export produces a document that (a) validates against the schema,
  (b) round-trips (re-import → identical structure). Landing this test is
  the P4b done-gate.

## Rationale

- **Standards positioning.** Owning a vocabulary is orthogonal to owning
  products. OSIF succeeds *for us* whether or not it's adopted by everyone;
  it succeeds *for the industry* only if it's neutral. So we make it neutral
  and hope for adoption. The pitch to buyers becomes "we scan for the same
  mechanics we defined in the open spec."
- **Doctrine coherence.** The completeness doctrine (ADR 0011) already
  enumerates classes internally. OSIF just publishes those enumerations
  under a CC0 license. The engineering cost is near-zero because the
  taxonomy already exists.
- **Benchmark backing.** The public benchmark (P4) scores tools against a
  corpus of defanged attacks. If each attack in that corpus is described in
  OSIF, the benchmark automatically encourages the industry to adopt OSIF
  when submitting new attack fixtures.

## Consequences

- We commit to keeping CAPABILITY-MAP in lockstep with OSIF's enumerations.
  Adding a new sink class means updating BOTH artifacts in one PR. This is
  enforced by the taxonomy-parity test (see above).
- We commit to running propose-invite-wait for community feedback BEFORE
  claiming adoption anywhere. The packet's honesty rule (§0.5) says: never
  claim adoption that hasn't happened.
- OSIF is a stable API. Breaking-schema changes are v-major bumps
  (`v0.1` → `v0.2` if we can accept small breakage during pre-1.0; `v1`
  once stable; `v2` requires a co-signed migration guide). All existing
  documents carry their `osif_version` so validators can dispatch.
- The `entry_point` and `capabilities_gained` enumerations MUST stay
  finite and PR-extensible. New enum values require a PR against `osif-spec`
  that includes at least one real-world example.

## Non-goals

- OSIF does not replace CVE, GHSA, or OSV. It is a *complement*. An OSIF
  document typically references its corresponding GHSA and one or more
  writeups.
- OSIF does not describe fixes or patches. That's OSV's job.
- OSIF does not carry executable payloads or live IOCs. Every `indicator`
  is defanged; the validator enforces this.
- OSIF is not versioned by vetlock's release cadence. Its own repo has its
  own tags.
