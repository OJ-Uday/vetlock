# vetlock red-team report

_2026-07-12 · vetlock v0.2.0 · adversarial-verify pass complete_

## Statistics

| Stage | Count |
|---|---|
| Total exploits proposed (6 red-teamers × 2 rounds) | **48** |
| After dedup (3 merged as duplicates) | **45** |
| **CONFIRMED** by independent adversarial verifier | **35** |
| PLAUSIBLE (verifier couldn't trace end-to-end) | 1 |

**Confirmation rate: 35/45 = 73%.** The verifier defaulted to REFUTED and required a full source-level trace to CLEAN or non-BLOCK. That the confirm rate is this high indicates the exploits were code-anchored, not speculative.

## Executive summary

Vetlock is not a reliable defense against a competent supply-chain attacker today. Six independent red-team teams generated 48 exploit candidates in disjoint territories; after de-duplication and an adversarial verifier that defaults to *refuted*, **35 of 45 candidates (73%) were confirmed end-to-end, 1 was plausible, 7 were partial, and only 2 were refuted**. That confirmation rate is not a sign that the red team was lenient — the verifier required a complete source-level trace from attacker input to CLEAN or non-BLOCK verdict — it is a sign that vetlock's detection surface has structural gaps at every layer: lockfile parsing, AST analysis, escalation logic, config trust, and cache lookup. Twelve of the confirmed exploits are severity **critical** (D1, D2, F1, F2, L1, L2, L3, L9, N1, N2, S1, S2, S10) and four more are compositional **BLOCK-severity** chains (C1, C2, C3, C5) that produce a fully clean verdict on genuinely malicious diffs.

An attacker aware of these findings would not attempt anything subtle. The three cheapest, highest-impact avenues, in order, are: (1) **ship the payload inside `bundledDependencies/*/node_modules/**` (N1)** — analyze.ts:116 unconditionally skips every `node_modules/` path, so 13 detectors go silent for zero effort; (2) **include a `.vetlock.json` in the same malicious PR (S2, C1, C2, D1, D2, D3)** — the config file is trusted from the diff under review and offers half a dozen documented knobs (`ignorePackages`, `severityOverride`, `ignorePathsInside: [""]`, `trustedPublishers: [""]`) that each silence the entire finding stream against the attacker's own package; and (3) **hand vetlock a pnpm-lock.yaml or yarn.lock (F1)** — the CLI advertises support but crashes with exit 3, which most CI configurations will treat as an infrastructure blip and not a security failure. None of these require any code obfuscation, AST cleverness, or novel evasion technique. They are configuration and format quirks.

## Cross-cutting themes

- **Detectors match literal AST shapes and string constants rather than semantic behavior.** Every process-env / require / child_process detector fires only on one canonical spelling of the operation, so trivially equivalent alternatives are invisible: destructured env (L1), spread env (L2), `Reflect.get(process.env, …)` (L5), computed alias `process["env"]` (L8), `require.call(null, 'child_process')` (L3), `eval("require('child_process')")` (L10), rot13-encoded URLs (L11), bare hostnames on unlisted TLDs (L4). This is not a small set of oversights — it is the structural cost of pattern-matching an untyped, dynamic language with a fixed list of syntactic templates. An attacker who spends thirty minutes with the visitor code can pick any equivalent form vetlock does not enumerate.

- **The `.vetlock.json` config file is trusted from the same diff it is meant to police.** S2 is the umbrella, and D1, D2, D3, C1, and C2 are five distinct ways the same trust boundary is monetised: overriding severity (D1), zero-prefix path ignores (D2), empty-string trust lists (D3), advisory-known package ignores (C1), and hardcoded evidence-file conventions that make one path prefix wipe an entire detector class (C2). Any red-team run against a tool that ships a config file will find variants of this — but the specific defect here is that string fields are accepted with no `.min(1)`, no wildcard restriction, and no round-trip semantic check against a benign baseline.

- **Escalation rules can be gamed by staying just under a co-occurrence threshold.** The escalation engine requires *combinations* of categories to promote WARN→BLOCK: OBF needs NET or INSTALL (S8, L9), typosquat needs a BLOCK-tier capability (D5), compound-3 needs both ≥3 findings *and* ≥2 categories (D4, S5). An attacker who ships OBF+EXEC+ENV+FS — the exact Shai-Hulud shape (L9) — never triggers the OBF→BLOCK rule because none of those four is NET or INSTALL. The rules encode analyst intuition about what a "real" attack looks like; the intuition is out of date.

- **Blindspots in name/module allowlists give free evasion by naming.** TOP_NPM_NAMES is missing openai, stripe, firebase, nodemailer, discord.js, and the entire `@sentry/*` and `@nestjs/*` scopes (D6). EXEC/NETWORK_MODULES is missing `inspector`, `trace_events`, `async_hooks` (N2) and `dns` (N3). URL_REGEX omits `.top`, `.pw`, `.zip`, `.click`, `.info`, `wss://`, and bare hostnames (L4, N3, L11). These are list-completeness bugs, but "add more entries" is not a fix — the underlying defect is that critical detection is gated on manually curated finite lists.

- **The analyzer conflates "what the lockfile says" with "what npm install actually installs."** F2 (link:true is skipped), F6 (yarn npm: alias hides real tarball name), N5 (git+ssh / patch: resolved fields are analyzed against registry content), C5 (case-preserved names miss lowercase TOP_SET / trust-store / advisory lookups), S9 (attacker-controlled file:// path), S10 (blank old integrity suppresses the tripwire), and S1 (integrity used as cache key before verification) are all instances of the same class: the parser trusts fields the attacker writes as ground truth for identity, path, or content.

## Priority recommendations

1. **Stop trusting `.vetlock.json` when it is part of the diff under review.** Load the config from the *before* side of the PR, or require it to live on a signed baseline branch, or reject any run where `.vetlock.json` appears in the changeset. Closes **S2, C1, C2, D1, D2, D3** (six confirmed criticals/BLOCKs). **Small** effort — one CLI flag, one existence check, one clear failure mode.

2. **Analyze `node_modules/**` inside tarballs.** Change `isAnalyzableSource` in analyze.ts:116 to include bundled node_modules — either recursively analyze them as first-class packages or, at minimum, run the capability detectors on their JS. Closes **N1** (single-shot 13-detector bypass, currently the cheapest full evasion). **Medium** effort — the extractor already walks these files; the guard is one line, but you will need per-file scoping for evidence.

3. **Make the pnpm/yarn code path actually work, or advertise npm-only and exit clean.** F1 (advertised-support crash) is a critical because CI treats exit 3 as flake, not fail. Fixing this properly means finishing the LockGraph shape for pnpm/yarn; a cheap intermediate is a `WARN unsupported-lockfile-format` finding with a non-zero-but-explicit exit code that CI can distinguish. Closes **F1**. **Medium** if fixed properly, **small** if the intermediate WARN path is acceptable.

4. **Rewrite the process/env/require detectors to bind at the *value* level, not the AST-node level.** Track the value flowing through variables and member access so `Reflect.get(process.env, x)`, `{...process.env}`, `const {NPM_TOKEN} = process.env`, `process["env"]`, and `require.call(null, 'child_process')` collapse to the same finding as the canonical form. Closes **L1, L2, L3, L5, L8, L10** (five criticals + one high). **Large** — this is real dataflow work in `capabilities.ts`, not a pattern addition, but each remaining literal-shape detector is another future exploit.

5. **Remove the co-occurrence gates on escalation rules where the individual signals are already high-signal.** OBF should escalate on OBF+EXEC or OBF+ENV or OBF+FS as well as OBF+NET/INSTALL (S8). Typosquat should escalate on any BLOCK-tier *or* multiple WARN-tier capabilities (D5). Compound-3 should escalate on 3+ WARN in any 1+ categories, dropping the ≥2 categories requirement (D4, S5). Closes **D4, D5, S5, S8, L9**. **Small** — these are threshold changes in `detectors/src/index.ts`, but they will require regression testing against known-benign packages.

6. **Fix the integrity-changed tripwire so an empty old integrity does not suppress it.** Treat `oldIntegrity === ''` on an existing name as "unknown-before, present-after" and emit the T5 BLOCK; separately, do not accept a lockfile-supplied integrity as a *cache key* before content verification. Closes **S1, S10** (two criticals) and hardens **C6** (plausible). **Small** for the T5 patch; **medium** for the cache-key fix, since the cache path has performance implications.

7. **Reject or hard-warn on `link: true`, yarn `npm:` aliases, git+ / patch: / file:// resolved fields whose apparent name diverges from the fetched name.** Analyze what will be *installed*, not what the lockfile *labels* it. Closes **F2, F6, N5, S9**. **Medium** — the fetchTarball path and lockfile parser both need to grow a "declared-vs-effective" identity concept.

8. **Reject empty-string entries and wildcard-adjacent values in every config field.** Schema-level `.min(1)`, reject `""` in `trustedPublishers` values (D3), `ignorePathsInside` values (D2), and any exact-match string that would degenerate to a match-all under `.includes` / `.startsWith`. Closes **D2, D3** and preempts the next generation of empty-string-degeneration bugs. **Small** — pure zod schema tightening.

9. **Fail-closed on analyzer errors and timeouts.** Currently `safeExtract` errors and >90s parses silently emit zero findings (S4), and unterminated backtick chains crash `@babel/traverse` (L6). A tarball that costs vetlock more than $X wall-clock or throws should emit a BLOCK-tier `analysis-failed` finding, not CLEAN. Closes **S4, L6** and the tail of similar future defects. **Small**.

10. **Match publisher trust-store entries with equality on parsed email components, not substring `.includes`.** Parse the email to `local@domain`, compare `domain` exactly (with an optional subdomain-suffix rule). Closes **S3**. **Small** — one function in `publishers.ts`.

## Methodology retrospective

The territory partitioning (Language / Format / System / Compositional / Defender-corner / Novel) worked well for coverage but leaked at the seams. Language and Format shared the AST-shape / lockfile-shape boundary cleanly; System covered cache and TOCTOU; Compositional was the highest-yield territory because it forced red-teamers to chain findings that individually looked minor (C1, C2, C3, C5 all combine 2–4 primitives into full-CLEAN outcomes). The Novel category earned its budget with N1, N2, N3, and N5 — genuinely orthogonal blindspots that would not have surfaced from a purely defensive read of the codebase. The one visible overlap was between Language (L11: rot13 URLs) and Novel (N3: DNS-subdomain exfil): both attack the "how does exfil leave the box" surface but from opposite ends of the pipeline; better dispatch would have kept them together. The one visible gap was **install-script surface** — no red-teamer meaningfully attacked the `install-script-added` / `preinstall`/`postinstall` detection path, and there is likely low-hanging fruit there.

The "default to refuted" verifier is doing its job. The 73% confirmation rate is not a signal of leniency — of the 10 non-confirmed outcomes, one was plausible, seven were partial (real defects with over-claimed impact), and only two were fully refuted. A lenient verifier produces almost no partials because it accepts the red-teamer's framing wholesale; the seven partials (F9, N4, F10, F8, C4, S7, S8) are exactly the "the mechanism is real but you overstated it" verdicts you want from a strict verifier. The two full refutations (F7 YAML-alias bomb, N6 pnpm peerDependencies) both turned on library-behavior claims that source inspection contradicted. The high confirm rate reflects that the red-teamers proposed *specific*, code-anchored exploits rather than speculative behaviors — a consequence of the six-team parallel structure and the pre-instruction to trace to a line number.

Three things I would change in a future run against a similar tool. First, **budget one team for install-hook / lifecycle-script exploits explicitly** — this was the ghost territory here. Second, **have the verifier flag partials by the axis they were over-claimed on** (severity, exploitability, reachability) rather than a single label; the seven partials collapsed useful distinctions. Third, **run the verifier at reduced effort as a first-pass filter and full effort only on the survivors** — the confirm rate suggests roughly two-thirds of verifier compute went to green-lighting well-researched claims, which is wasted intelligence when the goal is to catch the speculative minority.

## Confirmed exploits

### D1 — T5 tripwire (unconditional-BLOCK invariant) defeated by config: `severityOverride: {"integrity.hash-mismatch": "INFO"}` in .vetlock.json — same-version tarball swap now exits 0 (CLEAN)

🚫 **CONFIRMED** · **severity: critical** · confidence: confirmed · attacker effort: trivial (5-line .vetlock.json) · fix effort: small — exclude integrity.hash-mismatch (and other synthetic engine-emitted findings) from severityOverride & allowlist, OR treat it as an unconditional-BLOCK detector unaffected by user config

**Class:** escalation-abuse / config-override

**Description.** The synthetic integrity-changed finding is emitted by the ENGINE at packages/core/src/engine.ts:148-168 with detector id `integrity.hash-mismatch` and severity BLOCK. This is treated by the README/DETECTIONS.md as the T5 'unconditional BLOCK' invariant — a same-version tarball swap must always trip it. But that finding then flows through `applyConfig` in the CLI (packages/cli/src/cli.ts:208), which in packages/core/src/config.ts:111-115 does `const override = config.severityOverride[f.detector]; if (override) return { ...f, severity: override };`. There is NO exclusion for the synthetic integrity finding — the config's `severityOverride` treats `integrity.hash-mismatch` exactly like any other detector id. An attacker who lands a PR that (a) republishes foo@1.2.3 to point at a new (malicious) tarball with a fresh integrity, AND (b) commits a `.vetlock.json` with `{"severityOverride": {"integrity.hash-mismatch": "INFO"}}`, causes vetlock to emit the finding at INFO instead of BLOCK. The CLI's `computeVerdict` (cli.ts:236-244) then returns INFO — exit code 0. Distinct from F3 (drop-the-integrity-field bypass) because it lets the attacker leave BOTH lockfile sides fully integrity-populated while still neutering the tripwire; the .vetlock.json is the single source of the softening. Also distinct from S2's coarser ignorePackages / allowlist — S2 needs an explicit package match, D1 targets the DETECTOR id and thus neutralizes integrity checking universe-wide across every package in the diff.

**Fixture:**

```js
// .vetlock.json committed alongside the malicious lockfile change:
{
  "version": 1,
  "severityOverride": {
    "integrity.hash-mismatch": "INFO"
  }
}

// package-lock.json 'before' (already in main):
// "node_modules/foo": { "version": "1.2.3", "integrity": "sha512-AAA==", "resolved": "https://registry.npmjs.org/foo/-/foo-1.2.3.tgz" }

// package-lock.json 'after' (attacker's PR — SAME version, integrity flipped, tarball swap):
// "node_modules/foo": { "version": "1.2.3", "integrity": "sha512-EVIL==", "resolved": "https://registry.npmjs.org/foo/-/foo-1.2.3.tgz" }

// Reproduction (confirmed by direct simulation of the applyConfig chain):
//   node -e "const f={detector:'integrity.hash-mismatch',severity:'BLOCK'};
//            const cfg={severityOverride:{'integrity.hash-mismatch':'INFO'}};
//            if(cfg.severityOverride[f.detector]) f.severity=cfg.severityOverride[f.detector];
//            console.log(f.severity);"
//   => INFO
```

**Detectors bypassed:** `integrity.hash-mismatch`, `--fail-on (unless integrity.hash-mismatch explicitly listed)`

**Verifier's finding (confirmed).** End-to-end code path trace:

1. engine.ts:148-168 — when `change.kind === 'integrity-changed'` (same version, different integrity hash), the engine pushes a finding with `detector: 'integrity.hash-mismatch'`, `severity: 'BLOCK'`, `category: 'INTEG'`, and `evidence: [{ file: 'package.json', ... }]`. No special lock or protection is applied here; it is a plain Finding object.

2. cli.ts:191-205 — `runDiff` returns a `RunResult` containing that BLOCK finding.

3. cli.ts:208 — `result.findings = applyConfig(result.findings, config)` is called. The `config` is loaded from `.vetlock.json` via `loadConfig` at cli.ts:78.

4. config.ts:112-115 — inside `applyConfig`, the per-detector severity override `.map()` runs:
   ```ts
   const override = config.severityOverride[f.detector];
   if (override) return { ...f, severity: override };
   ```
   With `.vetlock.json` containing `"severityOverride": { "integrity.hash-mismatch": "INFO" }`, `override` equals `"INFO"` (truthy), and the finding is rewritten to `severity: 'INFO'`.

5. config.ts:35-43 — `ConfigSchema` defines `severityOverride` as `z.record(z.string(), SeveritySchema)`. The key type is `z.string()` with no constraints; `"integrity.hash-mismatch"` passes validation. `SeveritySchema` is `z.enum(['BLOCK','WARN','INFO'])` — `"INFO"` is a valid value. No `LOCKED_DETECTORS` set exists anywhere in the codebase.

6. cli.ts:210 — `result.verdict = computeVerdict(result.findings)` is called on the post-config findings. With the finding now at `INFO`, `computeVerdict` (cli.ts:236-244) returns `INFO` (or `CLEAN` if the INFO finding is dropped — but INFO does set `v = 'INFO'`).

7. cli.ts:228-231 — exit code logic: `BLOCK` → 2, `WARN` → 1, everything else → 0. Verdict `INFO` yields exit code 0.

The `--fail-on` guard (cli.ts:222-225) only helps if the CI recipe explicitly lists `integrity.hash-mismatch` in `--fail-on` or in `config.failOn`. The README's example does not include it, and the config-level `failOn` is also checked against `result.findings` AFTER `applyConfig` has already modified the findings' severities (though `isForcedFailure` matches on `f.detector`, not severity — however the `failOnCli` CLI flag path also checks `f.detector`, not severity, so that path would still fire if `integrity.hash-mismatch` is in `--fail-on`). Absent an explicit `--fail-on integrity.hash-mismatch`, CI exits 0 on a tarball swap.

**Concrete test:** Inline simulation matching the actual code logic:

const f = { detector: 'integrity.hash-mismatch', severity: 'BLOCK' };
const cfg = { severityOverride: { 'integrity.hash-mismatch': 'INFO' } };
const override = cfg.severityOverride[f.detector];  // 'INFO'
if (override) f.severity = override;                // rewrites to INFO
console.log(f.severity);  // 'INFO'

Then computeVerdict([{ severity: 'INFO' }]) returns 'INFO', and exit-code logic gives 0.

Full reproduction: commit a .vetlock.json with { "version": 1, "severityOverride": { "integrity.hash-mismatch": "INFO" } } alongside a package-lock.json where foo@1.2.3's integrity flips from sha512-AAA== to sha512-EVIL==. Run: vetlock diff before.json after.json; echo $? — outputs 0.

**Suggested fix.** Introduce a `LOCKED_DETECTORS = new Set(['integrity.hash-mismatch'])` and skip severityOverride/allowlist/ignore* rewrites for those detector ids; log a WARN on stderr when the user's config attempted to modify them so the softening attempt is visible.

### D2 — `ignorePathsInside: { "attacker-pkg": [""] }` — empty-string prefix silently drops every finding for the named package (any evidence path startsWith empty string)

🚫 **CONFIRMED** · **severity: critical** · confidence: confirmed · attacker effort: trivial · fix effort: trivial — enforce `.min(1)` on each prefix string in the zod schema; separately, treat an empty prefix as a schema error since it's meaningless

**Class:** config-injection self-exemption

**Description.** packages/core/src/config.ts:91-98 filters findings via `!ignore.some((prefix) => first.file.startsWith(prefix))`. `first.file` is the evidence-file path (e.g. `package.json`, `lib/index.js`, `.wasm`). In JavaScript, `'anything'.startsWith('')` is unconditionally true. The zod schema (`ignorePathsInside: z.record(z.string(), z.array(z.string()))`) does NOT enforce a min length on the prefix strings, so an empty-string entry passes validation. The result: a `.vetlock.json` containing `ignorePathsInside: { "attacker-pkg": [""] }` drops EVERY finding for `attacker-pkg` — including install.script-added, env.token-harvest, exec.new-module, fs.new-hotpath-write, bin.new-native-artifact, and even the synthetic integrity.hash-mismatch (whose evidence[0].file is `package.json`). Confirmed by direct simulation of the applyConfig chain — 2 BLOCK findings for a fake attacker-pkg were reduced to 0 findings. This is a stealthier variant of S2's ignorePackages: ignorePackages is a name-level exclusion that reviewers scanning `.vetlock.json` immediately recognize as suspicious ('why does the PR add an ignore for the package it's introducing?'); `ignorePathsInside` looks like a benign 'don't scan test/ folders' setting and hides the empty-string bomb in a legitimate-looking data structure. It also survives S2-style review lint rules that flag ignorePackages entries.

**Fixture:**

```js
// .vetlock.json committed with the malicious dep bump:
{
  "version": 1,
  "ignorePathsInside": {
    "legit-pkg-1": ["test/", "examples/"],
    "legit-pkg-2": ["docs/"],
    "attacker-pkg": [""]
  }
}

// Confirmed by:
//   const findings=[{detector:'exec.new-module',package:'attacker-pkg',severity:'BLOCK',
//                    evidence:[{file:'lib/index.js',line:1,snippet:'child_process'}]}];
//   const cfg={ignorePathsInside:{'attacker-pkg':['']}};
//   const kept=findings.filter(f=>{const ig=cfg.ignorePathsInside[f.package];
//     if(!ig||!ig.length)return true;const e=f.evidence[0]; if(!e)return true;
//     return !ig.some(p=>e.file.startsWith(p));});
//   => kept.length === 0
```

**Detectors bypassed:** `ALL — every finding for the named package is dropped before it reaches the verdict`

**Verifier's finding (confirmed).** The exploit is confirmed end-to-end. Two independent flaws combine:

1. Schema validation gap: `/Users/L122472/personal/vetlock/packages/core/src/config.ts` line 40 defines `ignorePathsInside: z.record(z.string(), z.array(z.string())).default({})`. There is no `.min(1)` on the per-package array's string entries, so `[""]` (an array containing an empty string) passes zod validation without error.

2. Filter logic unconditionally drops every finding: Lines 92-98 of config.ts run this filter:
   ```ts
   const ignore = config.ignorePathsInside[f.package];
   if (!ignore || ignore.length === 0) return true;
   const first = f.evidence[0];
   if (!first) return true;
   return !ignore.some((prefix) => first.file.startsWith(prefix));
   ```
   When `ignore = [""]`, `ignore.length === 0` is false (length is 1), so the early-return is skipped. `first` is guaranteed non-null because `validateFinding` (finding.ts line 205-210) enforces that `evidence` is non-empty and each entry has a non-empty `file` string. Then `first.file.startsWith("")` evaluates. In JavaScript, `String.prototype.startsWith("")` returns `true` for every string — including `'lib/index.js'`, `'package.json'`, `.wasm`, etc. So `ignore.some(...)` returns `true`, `!true` returns `false`, and the finding is filtered out.

Code-path trace for a BLOCK finding on `attacker-pkg`:
- Engine emits e.g. `{detector:'exec.new-module', package:'attacker-pkg', severity:'BLOCK', evidence:[{file:'lib/index.js', line:1, snippet:'child_process'}]}`
- `applyConfig` is called in the CLI (cli.ts applies config to findings before verdict)
- Filter step: `ignore = config.ignorePathsInside['attacker-pkg'] = [""]`; `ignore.length = 1` (not 0); `first.file = 'lib/index.js'`; `'lib/index.js'.startsWith('') === true`; finding is dropped
- ALL findings for `attacker-pkg` are dropped regardless of detector, category, or severity
- `computeVerdict` runs over zero findings for `attacker-pkg` — verdict is CLEAN or lower, exit 0

The exploit also correctly notes the social-engineering angle: `ignorePathsInside` in the existing codebase is documented as "glob-like path prefixes inside the package that should NOT be analyzed, e.g. `test/`, `examples/`". An entry like `{"legit-pkg-1": ["test/"], "attacker-pkg": [""]}` visually blends into a legitimate-looking config, unlike `ignorePackages` which is a more obviously suspicious package-level exclusion.

**Concrete test:** Set `ignorePathsInside: { "attacker-pkg": [""] }` in `.vetlock.json`. Run vetlock diff against a lockfile that adds `attacker-pkg` containing a lifecycle script (triggers `install.script-added` BLOCK) and a `require('child_process')` call (triggers `exec.new-module`). Without the config, verdict is BLOCK. With the config: applyConfig drops both findings because `'package.json'.startsWith('')` and `'lib/index.js'.startsWith('')` are both `true` in JavaScript. computeVerdict sees no remaining findings and returns CLEAN (exit 0).

**Suggested fix.** Change the schema to `ignorePathsInside: z.record(z.string(), z.array(z.string().min(1)))`. In applyConfig, additionally require the prefix to contain at least one path separator OR match a known directory to prevent single-character prefixes like `l` from also dropping everything under `lib/`. Best: emit a WARN on stderr for any ignore-prefix shorter than 2 chars.

### F1 — CLI advertises pnpm/yarn support; engine crashes with exit code 3 on any pnpm-lock.yaml or yarn.lock — no analysis performed

🚫 **CONFIRMED** · **severity: critical** · confidence: high · attacker effort: trivial — one commit swapping lockfile format, or a repo already on pnpm/yarn is auto-immune · fix effort: low — 4 lines in engine.ts, delete 10 lines in cli.ts, add a regression test that runs `diff pnpm-lock.yaml pnpm-lock.yaml` and asserts verdict is not a crash.

**Class:** silent-skip / crash-out via format mismatch

**Description.** The CLI diff subcommand runs parseLockfileText() as a preflight validator (cli.ts:181–187) that succeeds for pnpm-lock.yaml and yarn.lock, but then calls the engine's runDiff(beforeText, afterText, …), which internally executes JSON.parse(text) on the raw YAML text (engine.ts:77–78). runDiff has no format dispatcher — the TODO comment at cli.ts:178–180 admits this: 'runDiff still assumes npm'. So the second the tool sees a real pnpm-lock.yaml it throws 'Unexpected token l, "lockfileVe"... is not valid JSON', the CLI catches the error, writes it to stderr, and process.exit(3) is called. The README and CHANGELOG loudly announce pnpm/yarn support, and the CHANGELOG lists parsers for both as v0.2.0 shipped features — but no diff is ever produced against them. In a repo that has migrated to pnpm, vetlock is effectively bypassed for every dependency change: no CLEAN, no BLOCK — just crash. In the GitHub Action wrapper (action/action.yml:64–74) VETLOCK_EXIT=3 propagates as a failed status check that any maintainer with 'Bypass required status checks' can override, and even without bypass many teams route red-cross tool errors to 'known-broken, retry' triage. Attackers who want to blind vetlock in an npm-based repo can also weaponize this: land a benign-looking PR that swaps package-lock.json for a pnpm-lock.yaml, then land the malicious dep in a follow-up under the noise of tool errors.

**Fixture:**

```js
# Reproduce:
# 1. Save this file as /tmp/pnpm-lock.yaml (both 'before' and 'after' copies)
# 2. Run: vetlock diff /tmp/pnpm-lock.yaml /tmp/pnpm-lock.yaml
# Actual behavior: 'vetlock: Unexpected token l, "lockfileVe"... is not valid JSON', exit 3, no findings emitted.

lockfileVersion: '9.0'
importers:
  '.':
    dependencies:
      malicious-pkg:
        specifier: 1.0.0
        version: 1.0.0
packages:
  malicious-pkg@1.0.0:
    resolution:
```

### L1 — Env token harvest via `const { NPM_TOKEN } = process.env` destructuring bypasses env.token-harvest

🚫 **CONFIRMED** · **severity: critical** · confidence: confirmed · attacker effort: trivial · fix effort: moderate

**Class:** detector-evasion

**Description.** The env detector's MemberExpression visitor only fires for shapes matching `process.env.X`, `alias.X` (where alias is a processAlias), or `Object.<method>(process.env)`. Object-destructuring (`const { NPM_TOKEN } = process.env`) never creates a MemberExpression that visits — it uses ObjectPattern binding, which the visitor doesn't handle. So a package that steals NPM_TOKEN, GITHUB_TOKEN, AWS_*, and every other listed sensitive key via one destructuring statement emits ZERO envAccesses and env.token-harvest never fires. ```js

**Detectors bypassed:** `env.token-harvest`

**Verifier's finding (confirmed).** The VariableDeclarator visitor in capabilities.ts (line 568) begins with `if (!t.isIdentifier(path.node.id)) return;`. For `const { NPM_TOKEN, GITHUB_TOKEN, ... } = process.env`, the `id` node is an ObjectPattern, not an Identifier, so the visitor exits immediately without ever reaching the processAliases tracking logic (lines 600-618) or any other analysis.

The MemberExpression visitor (lines 501-554) does fire for the `process.env` subexpression (the `init` side of the declarator), but all three guarding conditions fail for this shape:
- `isProcessEnvDirect` checks `t.isMemberExpression(obj)` — false, because `obj` here is the bare `process` Identifier, not a nested MemberExpression.
- `isProcessEnvAlias` checks `processAliases.get('process') === 'process.env'` — 'process' has never been added to processAliases at this point, so false.
- `isProcessEnvViaProcessAlias` also requires `t.isMemberExpression(obj)` — false.

The `Object.keys` whole-object enumeration check (lines 540-554) requires the parent to be a CallExpression with callee.object `Object`; the parent here is a VariableDeclarator, so this also fails.

Consequence: `envAccesses` is `[]`. The env detector's `collectEnvKeys` finds nothing, `env.token-harvest` never fires (BLOCK suppressed). The `require('https')` on the fixture yields only a `net.new-module` WARN. Final verdict: WARN, not BLOCK — exactly as the exploit claims.

Code-path trace:
1. Parser produces: VariableDeclaration → VariableDeclarator { id: ObjectPattern([NPM_TOKEN,...]), init: MemberExpression(process.env) }
2. VariableDeclarator visitor: `t.isIdentifier(path.node.id)` → false → `return` at line 568.
3. MemberExpression visitor fires for `process.env`: obj=Identifier('process'), prop=Identifier('env'). isProcessEnvDirect=false, isProcessEnvAlias=false, isProcessEnvViaProcessAlias=false. No envAccess recorded.
4. env.ts iterates `f.envAccesses` — empty — no `env.token-harvest` finding generated.
5. engine.ts verdict loop: only WARN findings present → verdict = WARN.

**Concrete test:** Feed `'use strict';\nconst { NPM_TOKEN, GITHUB_TOKEN, AWS_ACCESS_KEY_ID } = process.env;` to `extractCapabilities('e.js', src, 'sha', src.length)` and assert `result.envAccesses.length === 0`. The VariableDeclarator visitor exits at line 568 of capabilities.ts because `id` is an ObjectPattern, not an Identifier, so no envAccess is ever pushed.

**Suggested fix.** In capabilities.ts, add an ObjectPattern visitor that inspects `path.node.id.properties` when `path.node.init` is `process.env` (or a processAlias). Record each destructured property name as an EnvAccess. Also visit VariableDeclarator when `id` is an ObjectPattern and `init` matches process/env aliases.

🚫 **CONFIRMED** · **severity: critical** · confidence: confirmed · attacker effort: trivial · fix effort: small

**Class:** detector-evasion

**Description.** The CallExpression visitor only recognizes `require('x')` when the callee is a bare Identifier named 'require'. Attackers can use `require.call(null, 'child_process')` or `require.apply(null, ['child_process'])`, where the callee is a MemberExpression (`require.call`), which never enters the require-detection branch. The module is loaded at runtime but vetlock sees ZERO execModules. Same trick works for `process.mainModule.require('child_process')`. Verified: `require.call(null, 'child_process')` produces execModules: []. A package that spawns processes this way is reported CLEAN by the exec detector.

**Fixture:**

```js
'use strict';
// Both of these load child_process at runtime; vetlock records nothing.

const cp2 = require.apply(null, ['child_process']);

const cp4 = module.constructor._load('child_process');
if (false) cp1.spawn('id');
```

**Detectors bypassed:** `exec.new-module`, `net.new-module (if the same trick loads http/https)`, `code.dynamic-loading-added`

**Verifier's finding (confirmed).** The CallExpression visitor in capabilities.ts (line 415) checks `t.isIdentifier(callee, { name: 'require' })`. For `require.call(null, 'child_process')` and `require.apply(null, ['child_process'])`, the callee is a MemberExpression (`require.call` / `require.apply`), not a bare Identifier, so this condition is false and the branch is never entered.

The alias-based fallback path (lines 462-465) checks `t.isMemberExpression(callee) && t.isIdentifier(callee.object)` and then looks up `moduleAliases.get(callee.object.name)`. For `require.call(...)`, `callee.object` IS an Identifier named `require`, but `require` is never stored in `moduleAliases` — that map only gets populated when `const x = require(...)` assignments are processed. So `moduleAliases.get('require')` returns `undefined`, and `noteModuleImport` is never called.

For `process.mainModule.require('child_process')`, the callee is `process.mainModule.require`. Its `callee.object` is `process.mainModule`, a MemberExpression, not an Identifier — so `t.isIdentifier(callee.object)` is false and the alias lookup is skipped entirely.

For `module.constructor._load('child_process')`, the callee is `module.constructor._load`. Its `callee.object` is `module.constructor`, also a MemberExpression — same result.

The VariableDeclarator visitor (line 580) for alias binding also checks `t.isIdentifier(init.callee, { name: 'require' })`, so `const cp = require.call(null, 'child_process')` similarly fails to register an alias.

Result: `execMods` stays empty for all four fixture variants. The exec detector (exec.ts) iterates over `f.execModules` — with that array empty, no BLOCK finding is emitted. The package reports as CLEAN (or at most WARN from an unrelated detector if network modules are also present).

**Concrete test:** Feed the fixture `const cp1 = require.call(null, 'child_process');` to `extractCapabilities('e.js', source, 'sha', source.length)`. Inspect the returned `execModules` array — it will be empty `[]`. Then confirm exec detector's `run(pair)` produces no findings because `pair.new.files[0].execModules` is empty.

**Suggested fix.** Recognize require.call / require.apply / require.bind — anywhere require is the .object of a MemberExpression called with the module name as an arg. Also recognize process.mainModule.require and module.constructor._load / Module._load — these are the documented ways to obtain require without a bare identifier.

### D3 — `trustedPublishers: { "chalk": [""] }` — empty-string trust list downgrades meta.maintainer-change takeover-BLOCK to INFO because `evidence.includes("")` is always true

🚫 **CONFIRMED** · **severity: high** · confidence: confirmed · attacker effort: trivial once you've compromised the publisher (which is a prerequisite for the underlying takeover attack) · fix effort: trivial — schema-enforce `z.string().min(3)` on each entry AND require `t.includes('@')` OR reject entries shorter than a plausible username

### L6 — Fail-soft-parse invariant violated: unterminated backtick chain crashes @babel/traverse with stack overflow

🚫 **CONFIRMED** · **severity: high** · confidence: confirmed · attacker effort: trivial · fix effort: small

**Description.** The 'Fail-soft parse — unparseable source → WARN, never crash' invariant is broken. ### N3 — DNS-based exfil channel — `dns.lookup()` / `dns.resolve()` are not in NETWORK_MODULES, and URL_REGEX doesn't match `wss://` or bare hostnames without a listed TLD, so stolen data can be leaked via DNS subdomain encoding with verdict CLEAN

Choice of `.example` (RFC-reserved) also gives the attacker plausible deniability during review. · fix effort: medium — (a) add `dns`, `node:dns`, `dns/promises`, `node:dns/promises` to NETWORK_MODULES so at minimum `net.new-module` fires; (b) recognize `dns.<lookup|resolve*|reverse>` calls with a computed / string-argument hostname as a `net.dns-exfil-candidate` WARN finding (subdomain-shaped, hostname > 30 chars, contains attacker-controlled variable inputs); (c) extend URL_REGEX to accept any dot-separated hostname whose right-most label is a valid TLD *or* an obviously-attacker-controlled `.example`/`.test`/`.invalid`/`.localhost` reserved TLD, and any `wss?://` scheme.

(c) Sign cache entries with a per-installation-secret HMAC and reject unsigned entries. (d) Empty/near-empty snapshots (files.length==0 && nativeArtifacts.length==0) should NEVER be returned from cache without re-fetch — legitimate packages have files.

### F9 — pnpm parser silently drops packages whose keys don't match the strict `name@version` shape — malicious dep hidden by key formatting alone

🟡 **PARTIAL** · **severity: high** · confidence: high · attacker effort: low — craft a lockfile with an unusual key shape; some private registries emit unusual keys naturally · fix effort: medium — 40 lines with a fallback URL-based name extractor plus regression tests

**Class:** silent-skip via strict key parse

**Description.** parsePnpmPackageKey (lockfile-pnpm.ts:171–181) requires the key match one of a handful of shapes: `/pkg@version`, `pkg@version`, or `pkg@version(peer@ver)`. If the last `@` is at position 0 OR missing, `at <= 0 → return null` and the package is silently skipped in the caller (`if (!parsed) continue;` in the two passes). Verified: `malicious-1.0.0` (dash instead of @), `(peer)malicious@1.0.0` (parens BEFORE name), `foo@` (empty version), `@1.0.0` (starts with @, no scope name) all return null. **Fixture:**

But in the current code it is unreachable via the CLI → runDiff path because pnpm lockfiles crash before reaching parsePnpmLockText in any analysis context. ### F10 — safe-extract totalBytes counter is racy across concurrent entries — attacker can push memory usage far beyond maxTotalBytes and DoS the analyzer

🟡 **PARTIAL** · **severity: medium** · confidence: medium · attacker effort: medium — craft an .tgz with 20+ 50MB-1 entries · fix effort: medium — 40 lines to refactor writeFileEntry into a streaming/backpressure-aware writer plus a fuzz test with 20+ entries just under the per-entry cap

**Class:** resource-exhaustion via async accounting race

**Description.** In safeExtract (extract.ts:88–231), each tar entry spawns an async writeFileEntry that buffers the ENTIRE entry's chunks into memory (`chunks: Buffer[]`) before calling fs.writeFile (lines 252–274). Only AFTER the write completes does hooks.addTotal(bytes) update totalBytes and check against maxTotalBytes (200–210). Because the parser dispatches entries concurrently and unbounded (there is no writer-side concurrency limit — inFlight just tracks in-flight promises for settleAll), an attacker who ships a tarball with N entries each just-under maxEntryBytes (50 MB - 1 byte) has all N entries buffered in RSS simultaneously. With maxTotalBytes = 500 MB and maxEntryBytes = 50 MB, 20 concurrent entries → 1 GB in memory before the accounting sees it. In a GitHub Actions runner with 7 GB RAM, a single malicious tarball can OOM the runner and stall the vetlock check. Worse: the tar library's own concurrency is chunk-driven — a slow-fake-write scenario where writeFile takes long lets more entries accumulate. The bytes-per-entry cap prevents ONE entry from being enormous, but the aggregate is not gated at ingest — only at write completion.

**Fixture:**

```js
// Conceptual reproduction (an actual malicious .tgz is too large to inline).
// Build a tarball with 20 file entries, each of size 50 MiB - 1 byte (below per-entry cap).
// e.g. via GNU tar:
//   for i in $(seq 1 20); do dd if=/dev/zero of=package/$i.js bs=1 count=$((50*1024*1024 - 1)); done
//   tar czf malicious.tgz package/
// Then pass malicious.tgz as the fetchOverride target.
//
// Direct trace path in the current source:
// 1. tar.Parser emits 20 'entry' events roughly simultaneously as chunks arrive.
// 2. Each entry spawns writeFileEntry, which buffers entry.on('data', ...) chunks into `chunks: Buffer[]`.
// 3. The check `if (bytes > limits.maxEntryBytes)` is per-entry — each entry is UNDER 50 MB, so it passes.
// 4. Total RSS across all 20 concurrent buffers = ~1 GB, but hooks.addTotal is only called after entry.on('end').
// 5. The maxTotalBytes:500 MB accounting fires AFTER — the memory is already allocated.
// 6. Node RSS peak → OOM in a small runner. vetlock aborts before completing analysis.
//
// Note: even a well-formed npm tarball rarely has 20 entries at exactly the per-entry cap, but a malicious
// package targeting vetlock users can exploit this. And under GitHub Actions default 7GB, it's realistic.
```

**Detectors bypassed:** `ALL — analyzer OOMs before extraction completes`

**Verifier's finding (partial).** The exploit's core claim — that `totalBytes` accounting in `safeExtract` happens post-entry rather than per-chunk, creating a window where multiple entries' data is in memory before the cap is enforced — is structurally correct but overstated in severity.

Actual code path in extract.ts:
- `writeFileEntry` (line 241-280) registers `entry.on('data', chunk => chunks.push(chunk))` and resolves only after `entry.on('end')`. `hooks.addTotal(bytes)` is called at line 273, AFTER the end event resolves the inner Promise.
- The tar parser (tar@7.5.20, read-entry.js line 38) starts every ReadEntry paused. When `writeFileEntry` calls `entry.on('data', ...)` (Minipass `on()` at minipass line 638-643), Minipass auto-resumes the stream. But `writeFileEntry` first awaits `fs.mkdir()` (line 251) before registering the data listener, so the entry stays paused until that async I/O completes.
- While Entry N is waiting on its `mkdir`, tar's CONSUMECHUNKSUB loop continues synchronously writing Entry N+1's body chunks into that entry's Minipass internal buffer (via `entry.write()` → `BUFFERPUSH`). This means Entry N's `chunks[]` (up to maxEntryBytes ≈ 50MB) and Entry N+1's Minipass internal buffer (up to 50MB) co-exist in memory before Entry N's `hooks.addTotal` has run.
- The PROCESSENTRY call for Entry N+1 fires synchronously inside Entry N's `end` handler chain (parse.js line 281: `entry.on('end', () => this[NEXTENTRY]())`), so Entry N+1's data can begin accumulating before the microtask queue runs Entry N's `hooks.addTotal`.

Why the exploit overstates the effect: `createReadStream` reads in 64KB chunks. The pipeline respects backpressure — when Minipass write() returns false (not-flowing), the parser signals not-ready, stalling the read stream. This limits how much data accumulates across entries before I/O yields. In practice, ~2-3 entries' worth of data can overlap, not 20. The "20 concurrent entries → 1 GB" scenario requires feeding all 1GB of compressed data in a single synchronous write to the parser, which the 64KB-chunk pipeline prevents.

What IS real: any single `writeFileEntry` call holds a full entry's decompressed chunks (up to 50MB) in `chunks[]` PLUS `Buffer.concat(chunks)` simultaneously at line 274 (2x per-entry peak). The totalBytes accounting fires after the entry finishes but BEFORE `fs.writeFile` completes — so two entries can overlap: one in `chunks[]` + concat buffer, one in Minipass internal buffer. That is ~150-200MB overhead above the intended cap, not 1GB.

The accounting-after-entry pattern (rather than per-chunk) is a genuine design flaw. The fix described (move accounting into the `data` handler) would close this gap. But the claim of "1GB RSS from 20 concurrent entries" requires conditions the Node.js event loop does not produce with a file-backed pipeline.

**Concrete test:** Build a 20-entry tarball (each entry 50MB-1 byte). Run safeExtract on it and measure Node.js RSS at peak. If the exploit's claim held, RSS would reach ~1GB before any totalBytes error. With backpressure in place, RSS peaks at ~150-200MB (2-3 entries overlapping), not 1GB. An additional test: move `hooks.addTotal(bytes)` into the `data` handler (before `chunks.push`) and rerun — RSS should stay bounded to maxEntryBytes + overhead.

**Refutation:** The exploit correctly identifies a real accounting race: `hooks.addTotal` fires at line 273 of extract.ts only after the entry's `end` event, not per-chunk. However, the claimed impact of 20 entries accumulating 1GB simultaneously is refuted by Node.js stream backpressure. `createReadStream` feeds data in 64KB chunks; Minipass/tar emit backpressure signals when entries are paused, stalling the pipeline. The realistic worst case is 2-3 entries overlapping (~150-200MB above cap), not 20 (1GB). The OOM-of-a-7GB-runner claim is implausible under normal I/O conditions.

**Suggested fix.** Cap concurrent in-flight writes at a small integer (e.g. 4) — a simple semaphore around writeFileEntry — and update totalBytes INSIDE the entry.on('data') handler on a per-chunk basis before buffering. Reject as soon as bytesReceivedAcrossAllEntries + chunk.length > maxTotalBytes. Stream chunks directly to disk instead of buffering into a Buffer[] — writeFile could then honor the per-entry cap as a truncate-and-fail.

🟡 **PARTIAL** · **severity: medium** · confidence: high · attacker effort: trivial — target a workspace-based monorepo repo · fix effort: medium — 25 lines in lockfile-pnpm.ts plus shortestPaths update, plus a corpus fixture pinning the workspace rollup output

**Class:** provenance blindness in monorepo layouts

**Description.** parsePnpmLockText only reads root deps from `importers['.']` OR from the flat top-level fields (lockfile-pnpm.ts:71–77). If a pnpm monorepo uses ONLY per-workspace importers (e.g. `importers['packages/app']`), the root node's `dependencies: []` stays empty. A malicious dep added via a workspace importer becomes an orphan in the graph: `nodes` and `byName` contain it, computeChangeset flags it as 'added', but shortestPaths(graph, 'node_modules/malicious') returns `[]` because there is no root→node path. Downstream: (a) the rollupByDirect (engine.ts:280–316) enumerates root's children — none — so the rollup summary is EMPTY and 'Impact by direct dependency' shows nothing, misleading a reviewer into thinking the change is dependency-free; (b) findings HAVE empty provenance, hiding WHICH direct dep pulled in the malicious code; (c) since typo detector fires only on ADDED packages against top names, it still fires for direct-visible names, but the impact-rollup silence is enough to reduce reviewer alarm. The ROOT of pnpm workspace lockfiles has to be all-importers-not-dot, so this is not just an edge case — it applies to Nx, Turborepo, and pnpm's own workspaces.

**Fixture:**

```js
# /tmp/pnpm-lock.yaml — no root importer, malicious dep only in packages/app
lockfileVersion: '9.0'
importers:
  packages/app:
    dependencies:
      malicious:
        specifier: 1.0.0
        version: 1.0.0
packages:
  malicious@1.0.0:
    resolution:
      integrity: sha512-badbadbad==

# Direct proof:
# $ node -e "const {parsePnpmLockText}=require('./packages/core/dist/lockfile-pnpm.js');
#   const {shortestPaths}=require('./packages/core/dist/lockfile.js');
#   const g=parsePnpmLockText(require('fs').readFileSync('/tmp/pnpm-lock.yaml','utf8'));
#   console.log('root deps:',g.nodes.get('').dependencies);
#   console.log('paths to malicious:',shortestPaths(g,'node_modules/malicious'));"
# root deps: []
# paths to malicious: []

# Then computeChangeset still emits an 'added' change, but provenance = [], rollup = {}.
```

**Detectors bypassed:** `provenance chains (invariant broken); rollup by direct dep (empty); reviewer-facing 'via' path`

**Verifier's finding (partial).** The exploit is accurate about the code paths but overstates the security impact. In `parsePnpmLockText` (lockfile-pnpm.ts:71-76), when no `importers['.']` key exists, the fallback reads `doc.dependencies/devDependencies/optionalDependencies` — all undefined in a workspace-only lockfile. So `collectRootDepNames` returns `[]`, and `rootNode.dependencies` stays `[]`. The malicious package IS still added to `nodes` and `byName` (first-pass at lines 96-117 correctly parses it). `computeChangeset` correctly emits an `added` Change for it. Detectors DO run via `opts.runDetectors(pair, change.name)` and produce findings. The verdict CAN still be BLOCK. However the two stated harms are confirmed: (1) `shortestPaths(newG, 'node_modules/malicious')` starts BFS from root `''` whose `dependencies=[]`, so it terminates immediately returning `[]`, meaning `f.provenance = []` for every finding on this package (engine.ts:174-181); (2) `computeRollup` reads `root.dependencies = []`, so `directNames` is empty, and `rollupByDirect = {}` — the "Impact by direct dependency" rollup is completely empty (engine.ts:280-316). The exploit does NOT produce CLEAN/WARN when BLOCK should fire — the security verdict is preserved. What is lost is provenance metadata and the direct-dep rollup, which hides the attribution trail from reviewers but does not suppress findings or change the verdict.

**Concrete test:** Fixture: pnpm-lock.yaml with only `importers['packages/app']` (no `importers['.']`) adding `malicious@1.0.0`. Trace: (1) `parsePnpmLockText` → rootImporter falls back to `{dependencies:undefined,...}` → `rootDepNames=[]` → `rootNode.dependencies=[]`; (2) `malicious@1.0.0` is added to nodes/byName correctly; (3) `computeChangeset` emits `{kind:'added',name:'malicious',...}`; (4) engine calls `opts.runDetectors(pair,'malicious')` → findings produced; (5) `shortestPaths(newG,'node_modules/malicious')` returns `[]` because BFS from root finds no children → `f.provenance=[]`; (6) `computeRollup` iterates `root.dependencies=[]` → `directNames=new Set()` → `rollupByDirect={}`. Result: verdict=BLOCK with findings, but provenance=[] and rollupByDirect={} on every finding.

**Refutation:** The exploit does not produce CLEAN/WARN when BLOCK is warranted. Detectors still receive the (old, new) snapshot pair for the malicious-added change and their findings still flow into the `findings` array, driving the verdict to BLOCK. The bypass is limited to provenance and rollup metadata, not the security verdict itself.

**Suggested fix.** Aggregate deps across ALL importers when building rootNode.dependencies, or synthesize a synthetic workspace-root node per importer and treat each as a root for provenance. Update shortestPaths to accept a set of roots. Add explicit provenance to the finding output showing which workspace introduced the dep — a reviewer reading 'via: packages/app → malicious' knows exactly which package to review.

### C4 — pnpm workspace-only importer + shortestPaths BFS via `root.dependencies` + rollupByDirect empty = malicious dep reports CLEAN in the direct-dep view even when detectors emit BLOCK

🟡 **PARTIAL** · **severity: BLOCK** · confidence: high · attacker effort: moderate — needs pnpm workspace-shape monorepo without a root package or PR that removes `importers['.']` · fix effort: small — importer-merge in the pnpm parser is 5-10 lines; rollup fallback is another handful

**Class:** compositional: monorepo lockfile shape + provenance BFS + rollup UI-truth channel

**Description.** Chain: (a) parsePnpmLockText (lockfile-pnpm.ts:71-77) reads root importer as `importers['.']` OR falls back to top-level flat `dependencies`/`devDependencies`/`optionalDependencies`. (b) If BOTH are absent (workspace-only project without a root package), rootDepNames=[] → rootNode.dependencies=[]. (c) shortestPaths in lockfile.ts:192-227 starts BFS from `''` and traverses `node.dependencies`. With root.dependencies=[], every provenance search returns [] — findings have provenance:[]. (d) computeRollup (engine.ts:280-316) enumerates ONLY children of root (line 285-292). With root.dependencies=[], directNames=Set() → rollupByDirect={} — completely empty. (e) The renderers (renderTTY, renderMarkdown, renderJSON) show the top-level 'Impact by direct dependency' as CLEAN or empty. (f) Individual findings STILL fire (they're per-package) but the reviewer-facing rollup, which vetlock's README highlights as the killer-feature reviewer touchpoint, is empty. On a real PR review, the reviewer's eye goes to the rollup FIRST; if it's empty, the diff looks like a routine transitive update. Compositional novelty over F8: F8 identified the provenance blindness, but this ALSO breaks the rollupByDirect signal — the PRIMARY reviewer-facing view — turning a BLOCK verdict into a review that looks like harmless upgrades. The attacker doesn't need to hide findings — they hide the ATTRIBUTION path.

**Fixture:**

```js
# pnpm-lock.yaml (attacker-crafted)
lockfileVersion: '9.0'

importers:
  packages/app:
    dependencies:
      '@types/typo-target':
        specifier: ^1.0.0
        version: 1.0.0

# NOTE: no importers['.'] and no top-level `dependencies:` — the pnpm parser
# reaches lockfile-pnpm.ts:72-76's fallback, which walks doc.dependencies (undefined).
# rootDepNames = [] → rootNode.dependencies = [] → shortestPaths+rollupByDirect are empty.

packages:
  '@types/typo-target@1.0.0':
    resolution: {integrity: sha512-attacker-controlled-hash===}
    dependencies:
      child-payload: 0.0.1

  child-payload@0.0.1:
    resolution: {integrity: sha512-attacker-hash2===}

snapshots:
  '@types/typo-target@1.0.0':
    dependencies:
      child-payload: 0.0.1
```

**Detectors bypassed:** `provenance chains (empty on every finding)`, `rollupByDirect (empty map)`, `reviewer-facing 'via' path in TTY/MD/JSON/SARIF renderers`, `any policy gate keyed off `Object.keys(rollupByDirect).length > 0``

**Verifier's finding (partial).** The underlying logic flaw in the pnpm parser is real: when a pnpm lockfile has `importers` but no `importers['.']` entry AND no top-level `dependencies`/`devDependencies`/`optionalDependencies` fields (workspace-only project), `parsePnpmLockText` (lockfile-pnpm.ts:71-77) falls back to a synthetic RawImporter with all three fields set to `undefined`. `collectRootDepNames` (line 162-168) skips all three undefined maps and returns `[]`. This makes `rootNode.dependencies = []`. In `computeRollup` (engine.ts:280-316), `root.dependencies` being empty means `directNames = new Set()` and `rollupByDirect = {}`. Likewise, `shortestPaths` BFS (lockfile.ts:192-227) starts from root with no edges, so every `provenance` array on findings would be `[]`.

HOWEVER, the exploit as described cannot be reproduced end-to-end via the CLI. The CLI at `cli.ts:176-180` calls `parseLockfileText` (lockfile-any.ts) as a preflight check for format detection and throws the parsed graph away. It then unconditionally passes the raw lockfile text to `runDiff` at cli.ts:191, which calls `parseLockfile(JSON.parse(oldLockfileText))` at engine.ts:77. For a pnpm YAML file, `JSON.parse` throws a `SyntaxError`. This exception propagates out of `runDiff`, is caught at cli.ts:201-204, and causes `process.exit(3)`. The CLI comment at line 178-180 even acknowledges: "runDiff still assumes npm" with a TODO to refactor. No detectors run, no rollup is computed, no findings are emitted — the tool crashes with exit code 3 before reaching any of the code paths the exploit claims to exploit.

The reproduction steps state "Confirm: findings array contains BLOCK for install.script-added" which is impossible under the actual code path. The exploit would only work if pnpm support were properly wired into `runDiff`, which it is not. The rollup blindspot is a latent bug in code that is not yet reachable via the public interface.

**Concrete test:** Run: `vetlock diff pnpm-lock.before.yaml pnpm-lock.after.yaml` where both files use the YAML format from the fixture. The tool will print a JSON parse error to stderr and exit with code 3, not exit 2 with BLOCK verdict and empty rollupByDirect. The pnpm parser (parsePnpmLockText) is only called by parseLockfileText for format detection at cli.ts:183, whose result is discarded; runDiff at cli.ts:191 re-parses using JSON.parse which cannot handle YAML.

**Refutation:** The CLI passes raw pnpm YAML text to `runDiff` which calls `JSON.parse` on it (engine.ts:77), throwing a SyntaxError before any detectors, rollup, or provenance code runs. Exit code is 3 (error), not 2 (BLOCK) with empty rollup. The exploit is unreachable via the described reproduction steps.

**Suggested fix.** (a) In parsePnpmLockText, when `importers` exists but `importers['.']` does not, MERGE all importers' deps into a synthetic root — the pnpm graph should always have a root that covers every workspace. (b) computeRollup should fall back to `Object.keys(newG.byName)` (or a curated subset) when rootDep is empty, so at least SOME grouping surfaces. (c) On empty rollupByDirect, the renderer should print a loud warning: 'lockfile has no root importer — provenance disabled; treat findings as top-level'.

### S7 — Advisory-range check uses `semver.satisfies(includePrerelease:true)` — a compromised republish at `ua-parser-js@0.7.29-x1` fails the exact-match range `=0.7.29`, so no GHSA cite is attached and the finding's confidence stays at default rather than being auto-upgraded to 'high'

🟡 **PARTIAL** · **severity: moderate** · confidence: high · attacker effort: trivial · fix effort: low

**Class:** escalation-abuse

**Description.** packages/detectors/src/advisories.ts:141-158 (advisoriesForVersion): `semver.satisfies(version, a.vulnerableRange, { includePrerelease: true })`. Ranges in the built-in ADVISORIES list are hand-written and mostly use `=<exact>` or narrow `>=x <=y`. semver's contract: `0.7.29-attacker` does NOT satisfy `=0.7.29` even with `includePrerelease: true` — `includePrerelease` controls whether prereleases can match, not whether they map to their base version. `>=1.95.5 <=1.95.7` does NOT match `1.95.5-x`. An attacker who publishes a compromised version with a prerelease tag (`ua-parser-js@0.7.29-p1`, `@solana/web3.js@1.95.5-hotfix`) causes no GHSA hit — the runAll() finding never receives the ` [GHSA: GHSA-fh58-9fw3-vf2v]` annotation and its confidence never gets bumped to 'high'. DETECTIONS.md prominently advertises GHSA correlation as a 'second-source citation, huge trust multiplier' — the user loses that citation on the exact attack shape that's most trivially engineered by an attacker who has read this document.

**Fixture:**

```js
// Corpus fixture: coa-rc-2021 with the version string changed from 1.2.9 to 1.2.9-hotfix.
// The advisory says: vulnerableRange: '=1.2.9 || =1.3.9 || =2.3.9'

// semver.satisfies('1.2.9-hotfix', '=1.2.9', { includePrerelease: true }) === false

// Finding for rc@1.2.9-hotfix (net.new-endpoint):
//   BEFORE fix:  "New network endpoint appeared: https://loader.example.invalid [GHSA: GHSA-g2q5-5433-rhrf]"
//                confidence: 'high'
//   AFTER attacker's prerelease trick:  "New network endpoint appeared: https://loader.example.invalid"
//                confidence: 'medium' (still, but no GHSA citation)

// User no longer sees the 'vetlock and GHSA agree' second source. The finding LOOKS more
// like a false positive than a corroborated attack. Reviewer more likely to dismiss.
```

**Detectors bypassed:** `GHSA advisory correlation (annotation + confidence upgrade)`

**Verifier's finding (partial).** The semver behavior is confirmed: `semver.satisfies('1.2.9-hotfix', '=1.2.9', { includePrerelease: true })` returns `false` (verified against the actual semver@7.8.5 module in the vetlock pnpm store). The same holds for every exact-match and bounded range in the ADVISORIES list against a prerelease version — `0.7.29-p1` fails `=0.7.29`, `1.95.5-hotfix` fails `>=1.95.5 <=1.95.7`.

Code path in `index.ts:139-149` (`runAll`): for each finding, `advisoriesForVersion(f.package, f.to ?? f.from)` is called. When the version is a prerelease (e.g. `1.2.9-hotfix`), `advisoriesForVersion` returns `[]`, the guard `if (matches.length === 0) continue` skips the annotation block, so neither the `[GHSA: ...]` message suffix nor the `f.confidence = 'high'` bump is applied.

However, the exploit overstates the impact. The content-based detectors (`net.new-endpoint`, `install.script-added`, `env.token-harvest`, etc.) operate on extracted tarball content, not on version metadata. A prerelease suffix does not affect what those detectors see; they still fire on the malicious payload and still produce BLOCK-tier findings. The overall verdict stays BLOCK.

For `net.new-endpoint` specifically, `net.ts:60` already emits `confidence: 'high'` on a changed package (pair.old !== null). The advisory correlation bump (`f.confidence = 'high'`) is therefore a no-op for that finding. The only real effect is loss of the `[GHSA: GHSA-fh58-9fw3-vf2v]` (or similar) citation text in the finding message.

The exploit does NOT produce CLEAN when it should be BLOCK, and it does not produce WARN when it should be BLOCK. It produces a BLOCK finding that is missing its corroborating citation and (for detectors that naturally emit confidence below 'high') misses the confidence upgrade. This is a real but narrower defect than claimed.

**Concrete test:** Publish `rc@1.2.9-hotfix` with the exact malicious payload from the coa-rc-2021 corpus fixture. Run `vetlock diff before.json after.json`. Observe: `net.new-endpoint` fires at BLOCK with `confidence: high` (already set by net detector for a changed package), message does NOT contain `[GHSA: GHSA-g2q5-5433-rhrf]`. Verdict is still BLOCK, exit code non-zero. For a detector that emits confidence 'medium' by default (e.g. `net.new-module` on an added package), the confidence stays 'medium' instead of being bumped to 'high' — that narrower form of the exploit is real.

**Refutation:** The claim that the prerelease trick meaningfully reduces confidence or enables dismissal is overstated for the primary attack path. `net.new-endpoint` on a changed package already emits `confidence: 'high'` by default (net.ts line 60), so the advisory bump is redundant. More critically, the exploit never achieves a CLEAN or WARN verdict — all BLOCK-tier content detectors fire regardless of version string format. The advisory correlation is an annotation layer only, not part of the verdict computation; its absence does not change the exit code or the BLOCK verdict.

**Suggested fix.** Normalize versions before the semver check: strip prerelease tag, then check exact/range match; if the base version matches a vulnerable range, still emit the citation with a note '[compromised prerelease of vulnerable base]'. Alternatively, when the advisory range uses `=` exact-match, ALSO test against `semver.eq(version, base)` after coercion.

### S8 — runAll's escalation rule 1 promotes OBF only when NET or INSTALL is present in the same package — an obfuscated payload paired with EXEC (child_process), ENV (env harvest), or FS (secret file read) does NOT escalate OBF from WARN to BLOCK

🟡 **PARTIAL** · **severity: moderate** · confidence: high · attacker effort: low · fix effort: trivial

**Class:** escalation-abuse

**Description.** packages/detectors/src/index.ts:93-104 (Escalation 1): the risky set is `f.category === 'NET' || f.category === 'INSTALL'`. Deliberately narrow: the classic worm signature is OBF + NET + INSTALL. But the ACTUAL BLOCK-tier categories are also EXEC, ENV, FS. An attacker who ships an obfuscated blob that (a) spawns a child_process from an unrelated obfuscated string (EXEC/BLOCK), (b) reads .npmrc (FS/BLOCK), (c) enumerates process.env (ENV/BLOCK) but does NOT import http/https/net and does NOT add a lifecycle script — has OBF/WARN + EXEC/BLOCK. The verdict is BLOCK, but the OBF finding stays WARN. So on a diff where the OBF blob is legitimate-looking (a bundler-produced minified output that also happens to encode the payload), a reviewer skimming for BLOCK-tier findings sees only the EXEC line and may think 'that's a build tool, fine' — missing the obfuscation signal that would tell them the blob is doing more than legit minification.

**Fixture:**

```js
// A version-diff that lands three findings:
//   1. obf.entropy-jump  (WARN, OBF)  — entropy jump in lib/util.min.js
//   2. exec.new-module   (BLOCK, EXEC) — imports child_process
//   3. fs.new-hotpath-read (BLOCK, FS) — reads ~/.npmrc
// Rule 1: risky set is {NET, INSTALL} → neither category present → OBF stays WARN.
// Rule 3: only 1 WARN → skipped.
// User's rendered TTY sees the obfuscation as a footnote WARN instead of a BLOCK; when
// noise is high, reviewers filter WARN in triage. The OBF signal is the exact tell that
// says 'the child_process call is doing more than what its variable names suggest'.

// Concrete diff: minified util.min.js goes from 2.1 bits/byte to 5.8 bits/byte (huge entropy
// jump). New require('child_process') statement. New readFile('~/.npmrc'). No new URLs,
// no lifecycle scripts.
```

**Detectors bypassed:** `obf escalation (OBF stays WARN under EXEC/ENV/FS pairing)`

**Verifier's finding (partial).** The exploit accurately identifies a real gap in Escalation Rule 1. In index.ts lines 93-104, the `risky` set is built exclusively from findings where `f.category === 'NET' || f.category === 'INSTALL'`. If a package only has EXEC/ENV/FS findings (no NET, no INSTALL), an OBF/WARN finding from that same package is never escalated to BLOCK. This is confirmed by reading the source:

- obf.ts always emits `severity: 'WARN'` for `obf.entropy-jump` and `obf.new-obfuscated-file`
- exec.ts emits `severity: 'BLOCK'`, category `'EXEC'`
- fs.ts emits `severity: 'BLOCK'`, category `'FS'`
- env.ts emits `severity: 'BLOCK'`, category `'ENV'`
- None of EXEC/ENV/FS appear in the `risky` set filter

However, the exploit does NOT produce CLEAN or WARN when it should be BLOCK. The exploit explicitly acknowledges: "Verdict is BLOCK (via EXEC/FS), but the OBF finding stays WARN." The overall verdict is correctly BLOCK because the EXEC and FS findings are natively BLOCK-tier. The attack is caught. The only failure is that the OBF finding is not co-escalated to BLOCK in the report, so a reviewer sees an OBF WARN alongside EXEC BLOCK rather than both at BLOCK level. The `detectors_bypassed` field only lists `obf escalation (OBF stays WARN under EXEC/ENV/FS pairing)`, not any primary detection capability. This is a presentation/correlation gap, not a detection bypass that causes the tool to report CLEAN or WARN on an attack it should BLOCK.

**Concrete test:** Scenario: package upgrades with (1) obf.entropy-jump in lib/util.min.js, (2) new require('child_process') — exec.new-module, (3) new fs.readFile('~/.npmrc') — fs.new-hotpath-read. No http/https/net imports, no lifecycle scripts. runAll produces: exec.new-module (BLOCK, EXEC), fs.new-hotpath-read (BLOCK, FS), obf.entropy-jump (WARN, OBF). Escalation 1 builds risky = {} (no NET or INSTALL findings) so OBF stays WARN. Escalation 3 requires 3+ WARN findings; only 1 WARN present, skipped. Final findings: two BLOCK + one WARN. Engine computes verdict = BLOCK. The tool correctly blocks the package; the OBF finding is not co-escalated to BLOCK but the overall verdict is not wrong.

**Refutation:** The exploit does not produce CLEAN or WARN for an attack that should be BLOCK. The overall verdict remains BLOCK because exec.new-module and fs.new-hotpath-write are inherently BLOCK-tier. The gap is purely that the OBF co-escalation rule is narrower than optimal (NET/INSTALL only, not EXEC/ENV/FS), so the obfuscation signal stays at WARN in the report rather than being promoted to BLOCK alongside the capability findings. A reviewer still sees a BLOCK verdict and BLOCK-tier findings; the OBF WARN is additional context that isn't promoted, not a missed detection.

**Suggested fix.** Expand rule 1's risky set from {NET, INSTALL} to {NET, INSTALL, EXEC, ENV, FS} — any BLOCK-tier capability category co-occurring with OBF should escalate. The threat model already treats these three as co-signals; there is no reason to trust obfuscation more when the payload happens to use child_process instead of http.
