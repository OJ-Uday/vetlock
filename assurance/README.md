# @vetlock/assurance — the standing adversary

**Not a shipped package.** This is the vetlock engine's continuous assurance harness. It
lives beside `packages/*` in the monorepo but is imported by no one; its job is to attack
`@vetlock/core` and prove that it (a) can't be taken down and (b) doesn't miss whole classes
of malicious behavior. See `~/personal/packets/PACKET-VETLOCK-ASSURANCE.md` for the mission
packet.

## What this package is (and isn't)

**Is:** a self-contained TypeScript workspace that spawns bounded, sandboxed engine runs,
feeds them adversarial inputs, and asserts a small shared oracle set (no-crash, no-hang,
no-OOM, fail-safe-verdict, finding-survives, identity-invariant). Findings become permanent
regressions in the vetlock corpus.

**Is not:** a place for functional malware. Every generated adversarial input is defanged —
payload endpoints resolve to `*.invalid`, secret-steal bodies are inert markers, exec
targets are `echo`-equivalents. A `assurance-defang-guard.test` scans generated corpora at
run time. If you're tempted to add a runnable exploit here, don't.

## Architecture (P0 slice)

```
assurance/
  docs/adr/     0001..0004 — harness-separate, four-modalities, bounded-sandboxed, run-tiers
  src/
    oracles/    the shared assertion set (framework-independent, pure functions)
    runner/     bounded/sandboxed engine runner (worker_threads + timeout + heap cap)
    report/     ASSURANCE.md generator (stamped, reproducible)
  test/         vitest self-tests for the harness itself
  report/       committed ASSURANCE.md (regenerated by `pnpm assurance:report`)
```

## Development

From the repo root:

```bash
pnpm --filter @vetlock/assurance install
pnpm --filter @vetlock/assurance build
pnpm --filter @vetlock/assurance test
pnpm --filter @vetlock/assurance assurance:report
```

## P0 status

Foundations only. No robustness/completeness generators yet — those are P1/P2. What P0
delivers: the oracle set, the bounded runner, an empty stamped report. The runner is
engine-agnostic (accepts any `AnalyzeFn`) so it can be exercised by synthetic scenarios
that deliberately hang / crash / OOM before P1 wires the real engine in.

## Branch protection (P5 release gate)

The `assurance-pr` workflow (`.github/workflows/assurance-pr.yml`) implements the
blocking PR tier from [ADR 0004](docs/adr/0004-run-tiers.md). For the release gate to
actually gate — "no vetlock release without green PR-tier assurance" — a repo
administrator must enable branch protection on `main` and mark **`PR-tier assurance
(blocking)`** (the job name) as a required status check.

Until that box is ticked, the workflow still runs on every PR and every push to `main`,
its badge on the root README turns red on regression, and reviewers can see the failure
in the checks tab. But GitHub itself won't refuse the merge. Enabling required-check
status is the last mile: it turns the automated adversary into a hard gate.

Companion workflow: `.github/workflows/assurance-scheduled.yml` runs the non-blocking
discovery tier nightly (fresh seed, larger budget, agent panel + differential adapters
when present) and opens an issue on any confirmed gap. Scheduled-tier findings must be
triaged and promoted into the PR-tier corpus before their issue closes — that's how the
blocking gate gets stronger over time without the PR tier itself getting slower.
